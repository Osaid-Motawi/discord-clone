# Phase 1 Data Model: Real-Time Chat & Video Calling

**Feature**: 001-realtime-chat-video | **Date**: 2026-07-14 | **Backend**: Convex

This model maps the spec's Key Entities to Convex tables in `convex/schema.ts`. Every access
pattern used by a query has a matching index (plan requirement + Principle III). Convex adds
`_id` and `_creationTime` to every document automatically; `_creationTime` is used for
chronological ordering where noted (so a separate `createdAt` is only added when a distinct or
mutable timestamp is needed).

Conventions:
- `Id<'table'>` denotes a Convex document reference.
- Timestamps are epoch milliseconds (`number`) unless they rely on `_creationTime`.
- Enum-like fields use `v.union(v.literal(...))`.

## Tables

### `users`
Profile data for each account. This is the **built-in `users` table provided by Convex Auth's
`authTables`** — we do **not** define it from scratch; we override the `authTables` `users`
definition to keep the fields we use (and add custom ones if ever needed). Auth credentials,
sessions, and verification live in the other `authTables` tables (`authSessions`,
`authAccounts`, `authVerifiers`, `authVerificationCodes`, `authRefreshTokens`,
`authRateLimits`) — managed by Convex Auth, not modeled here.

| Field | Type | Notes |
|-------|------|-------|
| `name` | `v.optional(v.string())` | Display name (FR-002). Enforce 1–32 chars in mutations. |
| `image` | `v.optional(v.string())` | **Avatar** URL/storage ref (FR-002). Built-in field — replaces the earlier `avatarUrl`. |
| `email` | `v.optional(v.string())` | From the Password provider; identity/display only. |

Other built-in `users` fields from `authTables` (`emailVerificationTime`, `phone`,
`phoneVerificationTime`, `isAnonymous`) are retained as-is and unused in v1.

**Indexes**: `email` (`email`) — the built-in index from `authTables` (used for
lookup/uniqueness).
**Relationships**: referenced by `serverMembers.userId`, `messages.authorId`, etc. The
authenticated id is obtained server-side via `getAuthUserId(ctx)` (see research.md §0/§2).
**Validation**: `name` non-empty, ≤ 32 chars (enforced in `users.updateProfile` and the
Password provider's `profile()` callback). **Note**: fields are optional at the schema level
(Convex Auth requirement); the app treats `name`/`image` as present after profile setup.

### `presence`
One row per user tracking liveness (Research §4).

| Field | Type | Notes |
|-------|------|-------|
| `userId` | `Id<'users'>` | Owner of this presence row. |
| `lastSeen` | `number` | Epoch ms of last heartbeat. |

**Indexes**: `by_user` (`userId`); `by_lastSeen` (`lastSeen`) — for stale sweep.
**Derived**: `isOnline = now - lastSeen < STALE_MS` (STALE_MS ≈ 30000). Pure function in
`src/lib/presence.ts`.
**Lifecycle**: upserted on heartbeat; stale rows treated as offline and swept.

### `servers`
A community with one owner and a reusable invite code (FR-005, FR-006).

| Field | Type | Notes |
|-------|------|-------|
| `name` | `string` | 1–64 chars (FR-005). |
| `imageUrl` | `string?` | Optional server image (FR-005). |
| `ownerId` | `Id<'users'>` | Creator/owner; not removable (Assumptions). |
| `inviteCode` | `string` | Current reusable invite token (FR-006). |

**Indexes**: `by_owner` (`ownerId`); `by_invite` (`inviteCode`) — join-by-link lookup.
**Relationships**: has many `channels`, `serverMembers`.
**State**: `inviteCode` regenerated by owner → old code becomes invalid (Assumptions). Deleting
a server cascades to its channels, messages, members, calls.

---

### `serverMembers`
Join table for User ↔ Server membership (FR-007, FR-008).

| Field | Type | Notes |
|-------|------|-------|
| `serverId` | `Id<'servers'>` | |
| `userId` | `Id<'users'>` | |
| `role` | `v.union('owner','member')` | Owner set at creation (FR-005). |

**Indexes**: `by_server` (`serverId`) — member list/sidebar; `by_user` (`userId`) — a user's
servers; `by_server_user` (`serverId`,`userId`) — membership check + dedupe.
**Validation**: at most one row per (serverId, userId). Removing a member deletes their row
(FR-009/FR-010); their authored messages remain (FR-010a).

### `channels`
Text or voice channel within a server (FR-011–FR-015).

| Field | Type | Notes |
|-------|------|-------|
| `serverId` | `Id<'servers'>` | |
| `name` | `string` | 1–64 chars. |
| `type` | `v.union('text','voice')` | (FR-013). |
| `isDefault` | `boolean?` | True for the auto-created "general" text channel (FR-011). |

**Indexes**: `by_server` (`serverId`) — channel list; `by_server_type` (`serverId`,`type`).
**Relationships**: text channel → many `messages`; voice channel → at most one active `call`.
**State**: created with server ("general", `type:'text'`, `isDefault:true`). Deleting a channel
cascades: delete its `messages` (FR-014) and end/clean any active `call` + `callParticipants`.

---

### `messages`
Text messages in a text channel (FR-016–FR-019, FR-016a, FR-010a).

| Field | Type | Notes |
|-------|------|-------|
| `channelId` | `Id<'channels'>` | Parent text channel. |
| `authorId` | `Id<'users'>` | Author (persists after author leaves — FR-010a). |
| `content` | `string` | 1–2000 chars (FR-016a). |
| `editedAt` | `number?` | Set on edit; presence of value ⇒ "edited" (FR-018). |

**Indexes**: `by_channel` (`channelId`) — ordered by `_creationTime` for newest-first
pagination (FR-019, SC-006).
**Ordering**: paginate on `by_channel` descending `_creationTime`.
**Validation**: `content` trimmed, 1–2000 chars. Edit/delete allowed only to `authorId`
(FR-018). Deleting removes the row for all (FR-018).

### `directMessageThreads`
A 1-on-1 DM conversation between two users who share a server (FR-021, FR-023).

| Field | Type | Notes |
|-------|------|-------|
| `userA` | `Id<'users'>` | Canonically the lexicographically smaller id. |
| `userB` | `Id<'users'>` | The other participant. |

**Indexes**: `by_pair` (`userA`,`userB`) — find-or-create dedupe (FR-023); `by_userA`
(`userA`), `by_userB` (`userB`) — list a user's DM threads.
**Validation**: store the pair in a canonical order (sorted) so `by_pair` guarantees a single
thread per unordered pair (FR-023). Eligibility (shared server) checked at creation (FR-021);
thread persists if servers later un-shared (Assumptions).

---

### `directMessages`
Messages within a DM thread; same rules as channel messages (FR-022).

| Field | Type | Notes |
|-------|------|-------|
| `threadId` | `Id<'directMessageThreads'>` | Parent thread. |
| `authorId` | `Id<'users'>` | |
| `content` | `string` | 1–2000 chars (FR-016a applies to DMs). |
| `editedAt` | `number?` | "edited" marker (FR-022). |

**Indexes**: `by_thread` (`threadId`) — newest-first pagination.
**Validation**: same as `messages`; edit/delete restricted to `authorId`.

---

### `typingIndicators`
Ephemeral "user is typing" rows scoped to a channel or DM thread (FR-020, Research §5).

| Field | Type | Notes |
|-------|------|-------|
| `scopeType` | `v.union('channel','dm')` | Where typing is happening. |
| `scopeId` | `string` | `channelId` or `threadId` (as string). |
| `userId` | `Id<'users'>` | Who is typing. |
| `updatedAt` | `number` | Refreshed while typing; stale ⇒ cleared. |

**Indexes**: `by_scope` (`scopeType`,`scopeId`) — active typers for a view; `by_scope_user`
(`scopeType`,`scopeId`,`userId`) — upsert/clear.
**Lifecycle**: upserted while typing (throttled), removed on stop or when
`now - updatedAt > TYPING_STALE_MS` (≈ 5000). The current user is excluded from their own view.

---

### `calls`
A live call bound to a voice channel or a DM thread (FR-024, FR-029, FR-030).

| Field | Type | Notes |
|-------|------|-------|
| `scopeType` | `v.union('channel','dm')` | Voice-channel call or DM call. |
| `channelId` | `Id<'channels'>?` | Set when `scopeType==='channel'`. |
| `threadId` | `Id<'directMessageThreads'>?` | Set when `scopeType==='dm'`. |
| `active` | `boolean` | False once the last participant leaves. |

**Indexes**: `by_channel` (`channelId`); `by_thread` (`threadId`); `by_active` (`active`) —
sweep/roster.
**State**: created on first join if none active for the scope; `active:false` when
`callParticipants` for it reaches zero (FR-028). Max 4 participants (FR-025).

---

### `callParticipants`
Who is connected to a call and their media state (FR-025–FR-029, FR-031, FR-032).

| Field | Type | Notes |
|-------|------|-------|
| `callId` | `Id<'calls'>` | |
| `userId` | `Id<'users'>` | |
| `micEnabled` | `boolean` | Mic on/off (FR-026, FR-027). |
| `cameraEnabled` | `boolean` | Camera on/off (FR-026, FR-027). |
| `lastSeen` | `number` | Heartbeat; stale ⇒ treated as left (FR-031). |

**Indexes**: `by_call` (`callId`) — roster + video tiles; `by_user` (`userId`) — enforce single
active call (FR-032); `by_call_user` (`callId`,`userId`) — upsert media state.
**Validation**: reject join when `by_call` count ≥ 4 (FR-025). On join, first remove any
existing row for `userId` (single active call, FR-032). Unexpected disconnect handled via
`lastSeen` staleness (FR-031). "Speaking" is derived client-side from audio levels (not stored).

---

### `signals`
WebRTC signaling payloads exchanged through Convex (Research §8).

| Field | Type | Notes |
|-------|------|-------|
| `callId` | `Id<'calls'>` | Call this signal belongs to. |
| `fromUserId` | `Id<'users'>` | Sender. |
| `toUserId` | `Id<'users'>` | Intended recipient (mesh is directed). |
| `kind` | `v.union('offer','answer','candidate')` | SDP or ICE. |
| `payload` | `string` | Opaque serialized SDP/ICE (justified external boundary). |

**Indexes**: `by_recipient` (`callId`,`toUserId`) — a peer reads signals addressed to it;
`by_call` (`callId`) — cleanup.
**Lifecycle**: recipient consumes then deletes its signals; orphaned rows from ended calls are
swept. `payload` is the single justified use of an opaque string/`any` boundary (Principle III
note).

## Cascade & integrity rules (enforced in mutations)

- **Delete channel** → delete its `messages`; end its `call` + `callParticipants` + `signals`.
- **Delete server** → delete its `channels` (with the above cascade), `serverMembers`, and the
  server's calls.
- **Remove member** → delete the `serverMembers` row; **keep** their `messages` (FR-010a);
  drop them from any of that server's active calls.
- **Join call** → remove any existing `callParticipants` row for the user first (FR-032);
  reject if target already has 4 participants (FR-025).
- **Leave / stale participant** → delete the `callParticipants` row; if the call has zero
  participants, set `calls.active=false` (FR-028, FR-031).
- **Message length** → all message-creating/editing mutations validate 1–2000 chars (FR-016a).
