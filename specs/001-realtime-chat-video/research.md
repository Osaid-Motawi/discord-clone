### Convex Auth — exact setup (labs.convex.dev/auth)

- **CLI init**: `npx @convex-dev/auth` — generates keys, sets env vars, scaffolds config.
- **Files**:
  - `convex/auth.ts` → `export const { auth, signIn, signOut, store, isAuthenticated } =
    convexAuth({ providers: [Password] })` — `convexAuth` from `@convex-dev/auth/server`,
    `Password` from `@convex-dev/auth/providers/Password`.
  - `convex/auth.config.ts` → `{ providers: [{ domain: process.env.CONVEX_SITE_URL,
    applicationID: "convex" }] }`.
  - `convex/http.ts` → `auth.addHttpRoutes(http)` on an `httpRouter()` from `convex/server`.
  - `convex/schema.ts` → spread `...authTables` (from `@convex-dev/auth/server`).
- **Env vars** (set via `npx convex env set`): `JWT_PRIVATE_KEY`, `JWKS` (both generated by
  init), and `SITE_URL` (e.g. `http://localhost:5173`). `CONVEX_SITE_URL` is provided by the
  deployment automatically.
- **React** (`@convex-dev/auth/react`): `ConvexAuthProvider` (replaces `ConvexProvider` in
  `src/main.tsx`); `useAuthActions()` → `{ signIn, signOut }` (call
  `signIn("password", formData)` with a hidden `flow` field of `"signIn"` | `"signUp"`);
  `useConvexAuth()` → `{ isLoading, isAuthenticated }`.
- **Server-side identity**: `getAuthUserId(ctx)` from `@convex-dev/auth/server` returns
  `Id<'users'> | null` (preferred typed helper; wraps `ctx.auth.getUserIdentity()`). Load the
  profile with `ctx.db.get(userId)`.

> **⚠ Impacts data-model.md.** `authTables` ships a **built-in `users` table** with optional
> fields `name`, `image`, `email`, `emailVerificationTime`, `phone`, `phoneVerificationTime`,
> `isAnonymous` (indexed on `email`, `phone`). We do **not** define our own `users` table from
> scratch — we override/extend the `authTables` `users` definition. **Avatar maps to the
> built-in `image` field** (rename our `avatarUrl` → use `image`); display name maps to `name`.
> Any extra fields (if needed) are added to the overridden `users` table and populated via the
> Password provider's `profile(params, ctx)` callback. `data-model.md` §`users` should be read
> with this mapping (`avatarUrl` ⇒ `image`).

### Convex React hooks (docs.convex.dev/client/react)

- `useQuery(api.mod.fn, args | "skip")` from `convex/react` → returns `undefined` while
  loading; live-subscribes and re-renders on data change; `"skip"` disables it.
- `useMutation(api.mod.fn)` from `convex/react` → stable async fn; optional
  `.withOptimisticUpdate((store, args) => …)` (`store.getQuery/setQuery`).
- `usePaginatedQuery(api.mod.fn, args, { initialNumItems })` from `convex/react` →
  `{ results, status, isLoading, loadMore(n) }`; `status ∈ { "LoadingFirstPage",
  "CanLoadMore", "LoadingMore", "Exhausted" }`. Server query takes
  `{ paginationOpts: paginationOptsValidator }` (`paginationOptsValidator` from
  `convex/server`) and calls `.order("desc").paginate(args.paginationOpts)` →
  `{ page, isDone, continueCursor }`. This is the current recommended infinite-scroll API.
- Server functions: `query` / `mutation` / `internalMutation` from `./_generated/server`,
  object form `{ args, returns?, handler }`; validators `v.*` from `convex/values`
  (`v.id("t")`, `v.union(v.literal(...))`, `v.optional(...)`, `v.any()`, …).
- Indexes: `defineTable({...}).index("by_x", ["a","b"])` (Convex appends `_creationTime` as the
  final sort key); read via `ctx.db.query("t").withIndex("by_x", q => q.eq("a", …))`.
- Cleanup jobs: `convex/crons.ts` with `cronJobs()` + `crons.interval("name",{minutes:1},
  internal.mod.fn)` (UTC), or one-shot `ctx.scheduler.runAfter/​runAt(...)` with `internal`
  from `./_generated/api`. Used for presence/typing/participant/signal staleness sweeps.

### WebRTC (MDN / W3C) — modern algorithm

- **Perfect Negotiation** is the current recommended pattern (supersedes the earlier
  "lower userId always creates the offer" note in §7/§8). Both peers run identical code; a
  per-connection `polite` flag resolves offer collisions. Assign deterministically in the mesh
  as `const polite = myUserId < remoteUserId;` (one polite + one impolite per pair, no
  round-trip). Use **no-arg `setLocalDescription()`** (auto offer/answer) and
  `setRemoteDescription(desc)` (implicit rollback for the polite peer). Key flags:
  `makingOffer`, `ignoreOffer`, `isSettingRemoteAnswerPending`. Events: `negotiationneeded`,
  `onicecandidate`, `ontrack`; property `signalingState`.
- **ICE restart**: call `pc.restartIce()` (no args) when `connectionState`/`iceConnectionState`
  becomes `"failed"`; it flows through `negotiationneeded` so perfect-negotiation rules apply.
  Treat `"disconnected"` as transient (debounce). Avoid legacy
  `createOffer({ iceRestart: true })`.
- **Connection monitoring**: prefer `connectionState` (aggregate ICE+DTLS) via
  `connectionstatechange` for failure/teardown decisions; `iceConnectionState` for ICE-specific
  diagnostics.
- **`iceServers`**: `[{ urls: "stun:stun.l.google.com:19302" }]` (STUN-only, v1). TURN entries
  add `username`/`credential` — omitted in v1 (documented NAT limitation, §7).
- **Media**: `navigator.mediaDevices.getUserMedia`, `pc.addTrack(track, stream)` →
  `RTCRtpSender`, `pc.ontrack`. **Mute via `track.enabled = false`** (no renegotiation; also
  drops the camera indicator); use `sender.replaceTrack(...)` for device switching — not
  `removeTrack` (which forces renegotiation).
- **Active speaker**: poll `RTCRtpReceiver.getSynchronizationSources()[].audioLevel`
  (0.0–1.0) for remote peers (lighter and more cross-browser than Web Audio on remote streams
  or `getStats()`); Web Audio `AnalyserNode` is fine for the local stream.

## 1. Backend & real-time data — Convex

- **Decision**: Use Convex as the sole backend: schema + queries (reactive reads) +
  mutations (writes). The React client subscribes with `useQuery`; any mutation that changes
  a queried result automatically re-pushes to all subscribers.
- **Rationale**: Satisfies Principle II (Real-Time Correctness) natively — the UI reflects
  server state through subscriptions with zero polling and no refresh. Removes the need for a
  separate WebSocket/Socket.io server (Principle I, Simplicity). End-to-end types flow from
  `schema.ts` through generated `api` into React (Principle III).
- **Alternatives considered**: Custom Node + Socket.io + Postgres (more moving parts, manual
  subscription bookkeeping, no end-to-end types); Firebase/Firestore (weaker relational
  querying and typing than Convex's typed schema/indexes).

## 2. Authentication — Convex Auth (password provider)

- **Decision**: Use `@convex-dev/auth@^0.0.94` (with `@auth/core@0.41.1` pinned) and the
  Password provider for signup/login. The authenticated identity is read server-side via
  **`getAuthUserId(ctx)`** (from `@convex-dev/auth/server`), wrapped in a `requireUser()`
  helper. Setup files/env vars/CLI are pinned in §0.
- **Rationale**: First-class Convex integration keeps auth checks inside every function
  (Principle IV) without a third-party auth service (Principle I). Password provider matches
  the spec's "sign up and log in" with no OAuth scope creep. `getAuthUserId` is the current
  typed helper (returns `Id<'users'>`), preferable to raw `ctx.auth.getUserIdentity()`.
- **Alternatives considered**: Clerk/Auth0 (extra dependency + external config, more than the
  spec needs); hand-rolled JWT/session auth (security risk, violates "security basics" by
  reinventing primitives); **Convex + Better Auth** (`@convex-dev/better-auth`) — now the more
  full-featured option Convex promotes, but heavier than this spec needs; Convex Auth stays the
  simpler choice (Principle I).
- **Verification note**: Convex Auth is a **beta** feature (pre-1.0). The `users` table comes
  from `authTables` (built-in `name`/`image`/`email`…); we extend it rather than defining our
  own, and **avatar maps to `image`** — see the ⚠ callout in §0.
- **Note**: Authorization (owner vs. member vs. author) is *application* logic layered on top
  of authentication, centralized in `convex/model/auth.ts`.

## 3. Authorization pattern — centralized guards

- **Decision**: Implement reusable server-side guards (built on `getAuthUserId(ctx)`):
  `requireUser(ctx)`,
  `requireMember(ctx, serverId)`, `requireOwner(ctx, serverId)`,
  `requireMessageAuthor(ctx, messageId)`, `requireCallAccess(ctx, callId)`,
  `requireDMParticipant(ctx, threadId)`. Every query/mutation calls the appropriate guard
  before touching data.
- **Rationale**: Principle IV mandates that every function validate authentication AND
  per-resource authorization. Centralizing prevents copy-paste drift and gives one auditable
  place for review (Principle VI testability — guards are unit-testable via `convex-test`).
- **Alternatives considered**: Inline checks per function (drift risk, harder to audit);
  a role/permission framework (over-engineered for owner-vs-member, violates Principle I).

## 4. Presence (online/offline) — heartbeat table + staleness threshold

- **Decision**: A `presence` table holds one row per user with `lastSeen` (ms). The client
  writes a heartbeat mutation every ~15s while connected and on focus/visibility changes. A
  user is "online" if `now - lastSeen < STALE_MS` (e.g., 30s). Cleanup of stale rows is done
  lazily on read and/or by a recurring `convex/crons.ts` job (`cronJobs()` +
  `crons.interval(...)`, calling an `internalMutation`) — verified current API (§0).
- **Rationale**: Meets SC-003 (< 5s visibility is achievable with a short heartbeat and
  threshold). Reads remain reactive via `useQuery` — the heartbeat is a *write* cadence, so
  Principle II (no polling for reads) holds. Presence derivation (`isOnline`) lives in
  `src/lib/presence.ts` as a pure, testable function (Principle VI).
- **Alternatives considered**: Convex WebSocket connection-state tracking (not a stable public
  API surface for this); third-party presence service (Principle I). Threshold values are
  documented constants so tests can assert the boundary.

## 5. Typing indicators — ephemeral heartbeat table

- **Decision**: A `typingIndicators` table stores `{ scopeType, scopeId, userId, updatedAt }`.
  The composer writes/refreshes a row while the user types (throttled, e.g., every 3s) and a
  `stopTyping` mutation (or staleness threshold ~5s) clears it. Members read active typers for
  the current channel/DM via `useQuery`.
- **Rationale**: Meets SC-004 (appears < 1s, clears within a few seconds). Same reactive-read /
  write-heartbeat model as presence (Principle II). Scope-keyed indexes keep reads O(active
  typers in scope).
- **Alternatives considered**: Broadcasting typing over a socket channel (needs separate
  server); storing typing on the message draft (leaks partial content — rejected).

## 6. Message history — newest-first pagination (infinite scroll)

- **Decision**: Store `messages` with a `channelId` + creation-time index; load the newest
  page first and fetch older pages as the user scrolls up, using Convex's paginated queries
  (`usePaginatedQuery` + server `paginationOptsValidator` and `.order("desc").paginate(...)`).
  A `useInfiniteMessages` hook owns the paging cursor and calls `loadMore(MESSAGE_PAGE_SIZE)`
  when `status === "CanLoadMore"`. (APIs verified in §0.)
- **Rationale**: Meets FR-019 (newest-first, infinite scroll) and SC-006 (next batch < 1s).
  Reactive: new live messages append through the subscription without breaking the scroll
  cursor. Indexed access pattern per Principle III / plan requirement.
- **Alternatives considered**: Load-all (fails at scale, poor perf); offset pagination
  (unstable under concurrent inserts — cursor pagination chosen instead).

## 7. Video/audio — native WebRTC, full-mesh, STUN-only

- **Decision**: Use the browser's native `RTCPeerConnection`. Each participant opens a direct
  peer connection to every other participant (full mesh). ICE uses Google public STUN
  (`stun:stun.l.google.com:19302`). No TURN in v1.
- **Rationale**: Meets FR-024–FR-031 for up to 4 peers. Full mesh at N≤4 means ≤3 connections
  per peer — acceptable and far simpler than integrating an SFU/SDK (Principle I). Mesh logic
  is isolated in `src/lib/webrtc/PeerMesh.ts` (a pure, testable seam — Principle VI).
- **Alternatives considered**: LiveKit/Twilio/Daily SDK (explicitly out per plan; extra deps +
  cost); SFU/MCU server (needed only beyond ~4–5 peers; over-engineered for v1).
- **Documented limitation (FR/plan)**: With STUN-only and no TURN, peers behind symmetric or
  strict NATs / restrictive firewalls may fail to establish a connection. This is a known,
  accepted v1 tradeoff and is surfaced to users as a connection failure message. TURN is a
  future addition. (MDN notes a meaningful fraction — commonly cited ~20–30% — of real-world
  connections need TURN; acceptable for a student v1 on permissive networks.)

### 7a. Negotiation & recovery — Perfect Negotiation + `restartIce()` (verified 2026-07-14)

- **Decision**: Use the **Perfect Negotiation** pattern (MDN) for every pairwise connection,
  not a hand-rolled "caller creates the offer" flow. Both peers run identical code; the
  `polite` role is assigned deterministically per connection as `polite = myUserId <
  remoteUserId`. Use argument-less `setLocalDescription()` (auto offer/answer),
  `setRemoteDescription(desc)` (implicit rollback for the polite peer), and the flags
  `makingOffer` / `ignoreOffer` / `isSettingRemoteAnswerPending` around the
  `negotiationneeded`, `onicecandidate`, and incoming-signal handlers.
- **Rationale**: This supersedes the earlier note in §8 that "the lower userId initiates the
  offer." Manual offer/answer ordering is race-prone under simultaneous renegotiation (adding
  camera mid-call, ICE restart). Perfect Negotiation is the MDN-canonical, collision-safe
  approach and keeps `PeerMesh.ts` a single code path (Principle I, VI). The userId comparison
  now assigns *politeness*, not offer initiation.
- **ICE restart**: on `connectionState === "failed"` (or `iceConnectionState === "failed"`)
  call `pc.restartIce()` — it routes through `negotiationneeded`, so the same
  perfect-negotiation code handles it. Treat `"disconnected"` as transient (debounce a few
  seconds before acting). Do **not** use the legacy `createOffer({ iceRestart: true })`.
- **Connection monitoring**: subscribe to `connectionstatechange` and drive teardown / UI off
  `connectionState` (aggregate ICE+DTLS); use `iceConnectionState` for diagnostics.
- **Media control**: toggle mic/camera with `track.enabled` (no renegotiation; camera
  indicator turns off); use `sender.replaceTrack()` only for device switching. Detect the
  active speaker via `RTCRtpReceiver.getSynchronizationSources()[].audioLevel` for remote
  peers (client-side; not stored — matches the data model's "speaking derived client-side").
- **Alternatives considered**: manual/ordered offer-answer (race-prone — rejected);
  `createOffer({iceRestart:true})` (legacy, composes poorly with `negotiationneeded`);
  Web Audio `AnalyserNode` on remote streams for speaking detection (cross-browser gaps vs.
  `getSynchronizationSources().audioLevel`).

## 8. WebRTC signaling — Convex `signals` table (no socket server)

- **Decision**: Exchange SDP offers/answers and ICE candidates by writing rows to a `signals`
  table (`{ callId, fromUserId, toUserId, kind, payload, createdAt }`) and subscribing via
  `useQuery`. Each peer reacts to signals addressed to it, feeds them into its
  `RTCPeerConnection`, and deletes/consumes them.
- **Rationale**: Reuses Convex's reactive subscription as the signaling channel, eliminating a
  Socket.io server (Principle I, II). Signaling payloads (SDP/ICE) are opaque strings stored
  as `v.string()`/`v.any()` at this single external boundary — the one justified place `any`
  may appear (Principle III note).
- **Alternatives considered**: Dedicated Socket.io/WS signaling server (extra infra, redundant
  given Convex realtime); manual copy-paste signaling (not usable).
- **Hygiene**: Consumed signals are deleted by the recipient; a staleness sweep removes orphan
  rows from abandoned calls to keep the table small.
- **Perfect-negotiation mapping (verified)**: the `kind` field distinguishes a **description**
  (`'offer'` | `'answer'`, carrying `pc.localDescription`) from a **`'candidate'`** (an ICE
  candidate). The recipient applies descriptions via `setRemoteDescription` and candidates via
  `addIceCandidate` per the §7a algorithm; an impolite peer that sets `ignoreOffer` also
  suppresses errors from that offer's candidates.

## 9. Single active call per user (FR-032)

- **Decision**: On join, the `joinCall` mutation first removes the user from any existing
  `callParticipants` row (leaving their previous call, updating that call's roster) before
  adding them to the new call.
- **Rationale**: Enforces FR-032 server-side so the invariant holds regardless of client state
  (Principle IV — not trusting the client). Keeps the "who's connected" indicator (FR-029)
  consistent.
- **Alternatives considered**: Client-side enforcement only (bypassable — rejected);
  allowing multiple calls (contradicts the clarified requirement).

## 10. Frontend styling & structure — Tailwind, no component library

- **Decision**: Build all UI as hand-written React components styled with Tailwind, using a
  dark Discord-like theme (server rail, channel sidebar, chat pane, member list). React Router
  handles navigation between auth, servers/channels, and DMs.
- **Rationale**: Principle I (no libraries beyond the plan; no UI kit). Small, purpose-built
  components keep the surface understandable. Theme tokens defined in `tailwind.config.ts`.
- **Alternatives considered**: MUI/Chakra/shadcn (adds a dependency and styling opinions the
  plan forbids); CSS-in-JS (unnecessary given Tailwind).

## 11. Testing strategy (Principle VI)

- **Decision**: Vitest + React Testing Library for components/hooks; `convex-test` for
  function-level authorization and behavior tests; two smoke tests exercising the critical
  flows end-to-end: **send a message** (mutation → subscription updates a second client view)
  and **join a call** (join mutation → roster + signaling wiring). Pure logic in `src/lib`
  (PeerMesh signaling reducer, presence/typing thresholds) gets focused unit tests.
- **Rationale**: Directly satisfies Principle VI (business logic separated from UI; critical
  flows get at least a smoke test). Authorization guards are unit-tested so security basics
  (Principle IV) are verified, not assumed.
- **Alternatives considered**: E2E browser tests with real media (flaky, heavy for v1 — mesh
  logic is instead tested via injected mock `RTCPeerConnection`).
